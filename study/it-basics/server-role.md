# 5. 서버의 역할

## 보이지 않는 곳에서 일어나는 일

지금까지 화면(2장), 요청과 응답(3장), 데이터의 형태(4장)를 살펴봤습니다.\
이제 그 데이터를 받아서 실제로 **처리하는 곳**을 들여다볼 차례입니다.

카페에서 주문이 들어오면 주방에서 음료를 만들듯, IT 서비스에서는 **서버**가 요청을 받아 처리합니다.\
서버는 사용자 눈에 보이지 않지만, 서비스의 핵심 역할을 담당합니다.

***

### 서버란 무엇인가요?

서버는 화면(클라이언트)의 요청을 받아 **처리하고 결과를 돌려주는 프로그램**입니다.

* 24시간 365일, 어딘가에서 항상 실행되고 있습니다
* 여러 사용자의 요청을 동시에 처리할 수 있습니다

서버는 다음과 같은 이름으로도 불립니다.

| 호칭                 | 의미                          |
| ------------------ | --------------------------- |
| **서버**             | 정식 명칭                       |
| **백엔드 (back-end)** | "뒷쪽"이라는 뜻으로, 사용자에게 보이지 않는 쪽 |

***

### 서버는 어떤 일을 하나요?

서버는 단순히 데이터를 전달하기만 하는 것이 아닙니다.\
요청을 받으면 **판단하고, 확인하고, 가공**하는 다양한 작업을 수행합니다.

할 일 목록(To-Do) 앱에서 사용자가 로그인을 시도하는 상황을 예로 들어 보겠습니다.

<table><thead><tr><th width="116.625">순서</th><th>서버가 하는 일</th></tr></thead><tbody><tr><td>1</td><td>이 이메일로 가입된 사용자가 있는지 데이터베이스에서 확인</td></tr><tr><td>2</td><td>비밀번호가 맞는지 검증</td></tr><tr><td>3</td><td>맞다면 → 사용자의 할 일 목록을 가져와 응답</td></tr><tr><td>4</td><td>틀리다면 → "비밀번호가 일치하지 않습니다" 오류 응답</td></tr></tbody></table>

이런 판단과 처리 과정을 **비즈니스 로직**이라고 부릅니다.

{% hint style="info" %}
**비즈니스 로직**이란 "이 서비스만의 업무 규칙"을 프로그램으로 옮겨 놓은 것입니다. 예를 들어 "비밀번호가 맞으면 로그인 허용, 틀리면 거부"도 하나의 비즈니스 로직입니다. 서비스의 핵심 기능이 모두 여기에 해당합니다.
{% endhint %}

비즈니스 로직은 서비스마다 다릅니다. 음악 서비스라면 "이 사용자가 프리미엄 회원인지 확인"하는 로직이 있을 것이고, 쇼핑 서비스라면 "재고가 남아 있는지 확인"하는 로직이 있을 것입니다.

***

### 서버는 어떻게 나를 알아보나요?

카페에서 단골 손님에게 적립 포인트를 주려면, 그 손님이 누구인지 알아야 합니다.\
서버도 마찬가지입니다. 요청을 보낸 사용자가 누구인지 확인하는 과정이 필요합니다.

이것을 **인증(Authentication)**&#xC774;라고 합니다.

{% hint style="info" %}
**인증**은 "당신이 누구인지 확인"하는 것이고, **권한**은 "확인된 당신이 무엇을 할 수 있는지 결정"하는 것입니다. 카페에서 멤버십 카드를 보여주는 것이 인증이라면, VIP 전용 메뉴를 주문할 수 있는지 확인하는 것이 권한입니다.
{% endhint %}

| 카페         | IT 서비스                | 설명                          |
| ---------- | --------------------- | --------------------------- |
| 멤버십 카드 제시  | **로그인**               | 아이디와 비밀번호로 본인을 확인           |
| 카드 뒷면의 바코드 | **토큰(Token)**         | 로그인 후 받는 인증표. 이후 요청마다 함께 전송 |
| VIP 전용 라운지 | **권한(Authorization)** | 인증된 사용자 중 특정 기능 사용 가능 여부    |

일반적인 흐름은 이렇습니다.

1. 사용자가 아이디/비밀번호로 **로그인**합니다
2. 서버가 확인 후 **토큰**을 발급합니다
3. 이후 요청할 때마다 이 토큰을 함께 보냅니다
4. 서버는 토큰을 보고 "이 사용자는 누구이고, 어떤 권한이 있는지" 판단합니다

{% hint style="info" %}
API를 호출할 때 **API Key**를 함께 보내는 경우도 있습니다. 이것은 "이 요청이 허가된 곳에서 온 것"임을 증명하는 열쇠와 같습니다.
{% endhint %}

### 서버는 어디에 있나요?

서버는 물리적인 컴퓨터 위에서 실행됩니다. 이 컴퓨터는 보통 **데이터센터**라는 시설에 모여 있습니다.

최근에는 직접 서버 컴퓨터를 운영하는 대신, **클라우드 서비스**(AWS, Google Cloud, Azure 등)를 이용하여 필요한 만큼 서버 자원을 빌려 쓰는 방식이 일반적입니다. 전기·수도를 직접 발전하지 않고 사용량만큼 요금을 내는 것과 비슷합니다.

{% hint style="info" %}
**클라우드**란 인터넷을 통해 서버, 저장 공간, 소프트웨어 등을 빌려 쓰는 서비스입니다. 직접 컴퓨터를 사서 관리할 필요 없이, 필요한 만큼만 사용하고 사용한 만큼만 비용을 내는 방식입니다. **AWS**(아마존), **Google Cloud**(구글), **Azure**(마이크로소프트)가 대표적인 클라우드 서비스입니다.
{% endhint %}

***

### 한 줄 정리

> 서버(백엔드)는 화면 뒤에서 요청을 처리하고, 데이터베이스와 연동하여 결과를 돌려주는 프로그램입니다. 사용자가 누구인지 확인하는 **인증**과, 어떤 기능을 쓸 수 있는지 판단하는 **권한** 관리도 서버의 중요한 역할입니다.

서버가 데이터를 처리한다는 것은 알았습니다.\
하지만 처리한 결과는 어디에 보관할까요? 다음 장에서는 **데이터베이스**를 살펴봅니다.
