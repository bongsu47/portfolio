# 3. 요청과 응답

## 요청하고, 응답받고

카페에서 "아이스 라떼 한 잔 주세요"라고 말하면, 주방에서 음료를 만들어 전달합니다.\
IT 서비스에서도 마찬가지입니다. 화면에서 버튼을 누르면 서버에 **요청**이 전달되고, 서버가 처리한 결과가 **응답**으로 돌아옵니다.

이번 장에서는 이 요청과 응답이 오가는 **통로(네트워크)**&#xC640; **규칙(API)**&#xC744; 살펴봅니다.

***

### 네트워크 : 주문이 오가는 길

네트워크는 화면(클라이언트)과 서버 사이를 **연결하는 통로**입니다. 인터넷도 네트워크의 한 종류입니다.

카페에 비유하면, 손님의 주문을 주방에 전달하고 완성된 음료를 다시 손님에게 가져다주는 **배달 서비스**와 같습니다.

* 화면이 서버에 "데이터를 보여줘"라고 요청하면 → 네트워크를 통해 전달됩니다
* 서버가 처리한 결과를 화면에 보내줄 때도 → 네트워크를 통해 전달됩니다

네트워크가 없으면 화면과 서버는 서로 소통할 수 없습니다. 배달원 없이는 음식을 받을 수 없는 것과 같습니다.

***

### API : 메뉴판과 주문 규칙

**API(Application Programming Interface)**&#xB294; 화면이 서버의 기능을 사용하기 위해 정해진 **약속**입니다.

{% hint style="info" %}
영어 약자가 어렵게 느껴질 수 있지만, 핵심은 간단합니다. API는 "서버야, 이런 일 해줘"라고 요청하는 **정해진 방법**입니다. 카페에서 메뉴판에 있는 음료만 주문할 수 있듯이, API에 정해진 요청만 보낼 수 있습니다.
{% endhint %}

카페의 **메뉴판과 주문 창구**에 비유할 수 있습니다.

* 메뉴판에 없는 음료는 주문할 수 없듯이, API에 정해지지 않은 요청은 보낼 수 없습니다
* "아이스 라떼 한 잔 주세요"처럼 정해진 형식으로만 주문하듯이, 요청에도 정해진 형식이 있습니다

| 카페               | IT 서비스                    |
| ---------------- | ------------------------- |
| 메뉴판              | API 문서 (어떤 기능이 있는지 정리한 것) |
| 주문 창구            | API 주소 (요청을 보내는 곳)        |
| "아이스 라떼 한 잔 주세요" | 요청 (Request)              |
| "주문하신 음료 나왔습니다"  | 응답 (Response)             |

***

### 요청에는 어떤 종류가 있나요?

화면이 서버에 할 수 있는 행동은 크게 네 가지입니다.\
할 일 목록(To-Do) 앱을 예로 들어 보겠습니다.

| 행동       | 의미            | To-Do 앱 예시           | 기술 용어       |
| -------- | ------------- | -------------------- | ----------- |
| **만들기**  | 새 데이터를 추가     | "장보기" 항목 추가          | POST        |
| **조회하기** | 저장된 데이터를 불러오기 | 할 일 목록 보기            | GET         |
| **수정하기** | 기존 데이터를 변경    | "장보기" → "마트 장보기"로 수정 | PUT / PATCH |
| **삭제하기** | 데이터를 제거       | 완료한 항목 삭제            | DELETE      |

{% hint style="info" %}
오른쪽의 POST, GET, PUT, DELETE는 **HTTP 메서드**라고 불리는 기술 용어입니다. 지금은 "만들기, 조회, 수정, 삭제에 각각 정해진 이름이 있구나" 정도만 알아두면 됩니다.
{% endhint %}

{% hint style="info" %}
이 네 가지를 **CRUD**라고 부릅니다 — Create(만들기), Read(조회하기), Update(수정하기), Delete(삭제하기)의 첫 글자입니다. 대부분의 IT 서비스는 이 네 가지 동작의 조합으로 이루어져 있습니다.
{% endhint %}

***

### 응답에는 어떤 정보가 담기나요?

서버가 요청을 처리한 뒤 돌려보내는 응답에는 **상태 코드**가 포함됩니다.\
택배 배송 상태에 비유하면 이해하기 쉽습니다.

| 상태 코드   | 의미      | 택배 비유              |
| ------- | ------- | ------------------ |
| **2XX** | 성공      | "배송이 완료되었습니다"      |
| **4XX** | 요청 쪽 문제 | "주소가 잘못되었습니다"      |
| **5XX** | 서버 쪽 문제 | "물류센터에 문제가 발생했습니다" |

{% hint style="info" %}
**상태 코드**는 세 자리 숫자로 이루어져 있습니다. 앞자리만 보면 대략적인 의미를 알 수 있습니다. 2로 시작하면 성공, 4로 시작하면 요청한 쪽의 실수, 5로 시작하면 서버 쪽의 문제입니다.
{% endhint %}

예를 들어,

* 할 일 목록을 성공적으로 불러왔다면 → **200** (성공)
* 존재하지 않는 항목을 조회했다면 → **404** (찾을 수 없음)
* 서버에 예상치 못한 오류가 발생했다면 → **500** (서버 오류)

***

### 한 줄 정리

> 화면(클라이언트)은 **네트워크**를 통해 서버에 **요청**을 보내고, 서버는 처리 결과를 **응답**으로 돌려줍니다. 이 약속을 **API**라고 부르며, 대부분의 서비스는 만들기·조회하기·수정하기·삭제하기(CRUD) 네 가지 동작으로 이루어져 있습니다.

요청과 응답이 어떻게 오가는지 알았습니다.\
