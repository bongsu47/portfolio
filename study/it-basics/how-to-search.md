# 8. 정보를 찾는 방법

구글에서 "맛집"을 검색하면 순식간에 수백만 개의 결과가 나타납니다.\
네이버에서 "날씨"를 검색하면 오늘의 기온이 바로 보입니다.

우리는 매일 검색을 사용하지만, 그 안에서 어떤 일이 벌어지는지는 잘 모릅니다.\
이번 마지막 장에서는 검색이 어떻게 작동하는지, 그리고 최근 AI 기술과 만나 어떻게 진화하고 있는지를 살펴봅니다.

***

### 검색엔진은 어떻게 작동하나요?

구글, 네이버 같은 **검색엔진**은 인터넷에 있는 수십억 개의 웹 페이지 중에서 원하는 정보를 찾아주는 서비스입니다.

도서관에서 사서가 책을 찾아주는 과정에 비유하면 이해하기 쉽습니다.

<table><thead><tr><th>도서관 사서</th><th width="118.8203125">검색엔진</th><th>설명</th></tr></thead><tbody><tr><td>새로 들어온 책을 모두 확인</td><td><strong>크롤링</strong></td><td>인터넷의 웹 페이지를 돌아다니며 내용을 수집</td></tr><tr><td>목록 카드를 만들어 분류</td><td><strong>인덱싱</strong></td><td>수집한 페이지를 정리하여 색인(목록)을 만듦</td></tr><tr><td>요청에 맞는 책을 찾음</td><td><strong>매칭</strong></td><td>검색어와 일치하는 페이지를 색인에서 찾음</td></tr><tr><td>가장 관련 높은 책부터 건넴</td><td><strong>랭킹</strong></td><td>관련도가 높은 순서대로 결과를 정렬</td></tr></tbody></table>

이 과정에서 핵심은 **키워드 매칭**입니다.

"서울 맛집"을 검색하면, 검색엔진은 "서울"과 "맛집"이라는 **단어가 포함된** 페이지를 찾습니다.\
단어의 뜻을 이해하는 것이 아니라, 글자가 일치하는지를 확인하는 방식입니다.

{% hint style="info" %}
**키워드 검색**은 "이 단어가 문서에 있느냐 없느냐"를 기준으로 결과를 찾는 방식입니다. 빠르고 정확하지만, 같은 뜻의 다른 표현은 찾지 못하는 한계가 있습니다.
{% endhint %}

***

### 키워드 검색의 한계

키워드 검색은 빠르고 효과적이지만, 한계도 있습니다.

#### 같은 뜻, 다른 단어

"자동차 수리"를 검색하면, "차량 정비"라고 쓰인 문서는 놓칠 수 있습니다.\
사람은 "자동차 수리"와 "차량 정비"가 같은 뜻이라는 것을 알지만, 키워드 검색은 **글자**만 비교하기 때문입니다.

| 검색어      | 찾을 수 있는 문서     | 놓치는 문서         |
| -------- | -------------- | -------------- |
| "자동차 수리" | "자동차 수리 비용 안내" | "차량 정비 서비스 소개" |
| "감기 치료"  | "감기 치료법 5가지"   | "독감 예방과 관리"    |

#### 의도를 이해하지 못함

"비 오는 날 실내에서 할 만한 것"을 검색하면, 사람은 "영화관", "볼링장", "방탈출 카페" 등을 떠올립니다.\
하지만 키워드 검색은 이 문장의 **의도**를 파악하지 못하고, "비", "실내" 같은 단어가 포함된 문서만 찾습니다.

***

### 의미를 이해하는 검색

이런 한계를 넘어서기 위해 등장한 것이 **의미 기반 검색**입니다.\
단어의 **글자**가 아니라 **뜻**을 비교하는 방식입니다.

| 비교             | 키워드 검색         | 의미 기반 검색    |
| -------------- | -------------- | ----------- |
| 비교 대상          | 글자(단어)         | 의미(뜻)       |
| "자동차"와 "차량"    | 다른 단어 → 매칭 안 됨 | 비슷한 뜻 → 매칭됨 |
| "강아지"와 "puppy" | 다른 단어 → 매칭 안 됨 | 같은 뜻 → 매칭됨  |

이것이 가능한 이유는 **임베딩(Embedding)**&#xC774;라는 기술 덕분입니다.

{% hint style="info" %}
**임베딩**이란 텍스트를 **숫자 목록**으로 변환하는 기술입니다. 뜻이 비슷한 단어는 비슷한 숫자로 변환되기 때문에, 숫자끼리 비교하면 의미가 통하는 문서를 찾을 수 있습니다. 마치 모든 언어를 하나의 "좌표"로 바꿔서, 좌표가 가까우면 뜻이 비슷한 것으로 판단하는 방식입니다.
{% endhint %}

예를 들어 봅시다.

| 텍스트     | 변환된 숫자 (단순화)        | 거리               |
| ------- | ------------------- | ---------------- |
| "강아지"   | \[0.9, 0.1, 0.8]    | —                |
| "puppy" | \[0.88, 0.12, 0.79] | 매우 가까움 (= 비슷한 뜻) |
| "냉장고"   | \[0.1, 0.7, 0.2]    | 매우 멀음 (= 다른 뜻)   |

"강아지"를 검색하면, 글자가 전혀 다른 "puppy"도 가까운 의미로 함께 찾아지는 것입니다.

***

### AI와 검색의 만남 : RAG

의미 기반 검색까지 발전했지만, 여전히 한 가지 아쉬운 점이 있습니다.\
검색 결과는 "관련 문서의 목록"일 뿐, 사용자가 원하는 **답**을 직접 만들어주지는 않습니다.

여기서 등장하는 것이 **AI(인공지능)**&#xC785;니다.

{% hint style="info" %}
**AI(인공지능)**&#xB294; 사람처럼 생각하고 판단할 수 있도록 만들어진 프로그램입니다. 그중에서도 LLM(대형 언어 모델)은 방대한 양의 텍스트를 학습하여 사람의 말을 이해하고 자연스러운 문장으로 답변할 수 있습니다. ChatGPT, Claude 같은 서비스가 대표적인 LLM입니다.
{% endhint %}

LLM은 매우 똑똑하지만 한계가 있습니다.

<table><thead><tr><th width="258.578125">한계</th><th>설명</th></tr></thead><tbody><tr><td><strong>학습 시점 이후의 정보를 모름</strong></td><td>2024년까지 학습했다면, 2025년 소식은 답할 수 없음</td></tr><tr><td><strong>내부 문서를 모름</strong></td><td>우리 회사의 사내 매뉴얼, 제품 설명서 등은 학습하지 않았음</td></tr><tr><td><strong>환각(Hallucination)</strong></td><td>모르는 것도 그럴듯하게 지어내는 경우가 있음</td></tr></tbody></table>

{% hint style="info" %}
**환각(Hallucination)**&#xC774;란 AI가 마치 사실인 것처럼 틀린 정보를 답하는 현상입니다. 예를 들어 "이순신 장군은 1595년에 한글을 발명했습니다"처럼 자신감 있게 말하지만 완전히 틀린 내용을 생성할 수 있습니다.
{% endhint %}

이 문제를 해결하기 위해 **검색과 AI를 결합**한 방식이 등장했습니다.\
바로 **RAG(Retrieval-Augmented Generation)**&#xC785;니다.

#### RAG는 어떻게 작동하나요?

RAG는 한마디로 **오픈북 시험**과 같습니다.\
시험을 볼 때 교과서를 펼쳐놓고 관련 내용을 찾아 읽은 뒤, 그 내용을 바탕으로 답안을 작성하는 것입니다.

| 오픈북 시험             | RAG                          |
| ------------------ | ---------------------------- |
| 문제를 읽는다            | 사용자의 **질문**을 받는다             |
| 교과서에서 관련 부분을 찾는다   | 문서 저장소에서 관련 내용을 **검색**한다     |
| 찾은 내용을 참고하여 답안을 쓴다 | 검색 결과를 참고하여 AI가 **답변을 생성**한다 |

실제 과정을 단계별로 보면 이렇습니다.

1. 사용자가 질문합니다: "우리 회사 연차 신청 방법이 어떻게 되나요?"
2. 시스템이 회사 문서 저장소에서 관련 내용을 **검색**합니다 (의미 기반 검색)
3. 검색된 문서를 AI에게 함께 전달합니다: "이 문서를 참고해서 답변해 줘"
4. AI가 문서 내용을 읽고, 자연스러운 문장으로 **답변을 생성**합니다

> "연차 신청은 사내 포털 → 인사관리 → 휴가신청에서 가능합니다. 최소 3일 전에 신청해야 하며, 팀장 승인이 필요합니다."

검색만 했다면 "연차 규정.pdf"라는 파일 이름만 보여줬을 것입니다.\
RAG 덕분에 AI가 문서를 읽고 **사용자가 원하는 형태로 답변을 직접 만들어** 줍니다.

{% hint style="info" %}
**RAG**의 핵심은 AI가 "아는 것"이 아니라 "찾아본 것"을 바탕으로 답한다는 점입니다. 덕분에 환각이 줄어들고, 회사 내부 문서처럼 AI가 학습하지 않은 정보에 대해서도 정확한 답변이 가능해집니다.
{% endhint %}

### 하이브리드 검색 : 두 가지 장점을 합치다

지금까지 키워드 검색과 의미 기반 검색을 따로 살펴보았습니다.\
그런데 실제 서비스에서는 이 **둘을 함께** 사용하는 경우가 많습니다.\
이것을 **하이브리드 검색**이라고 합니다.

왜 둘을 합칠까요? 각각 잘하는 것이 다르기 때문입니다.

| 상황                      | 키워드 검색    | 의미 기반 검색                |
| ----------------------- | --------- | ----------------------- |
| "MISO-4022 오류" (정확한 코드) | 정확히 찾음    | 비슷한 다른 코드가 섞일 수 있음      |
| "시스템이 느려지는 문제" (의미)     | "느려짐"만 찾음 | "성능 저하", "응답 지연"도 함께 찾음 |

"MISO-4022"처럼 **정확한 단어**가 중요한 경우에는 키워드 검색이 유리하고, "시스템이 느려지는 문제"처럼 **의미**가 중요한 경우에는 의미 기반 검색이 유리합니다.

하이브리드 검색은 두 방식의 결과를 합산하여, **정확한 단어 매칭과 의미 매칭을 동시에** 잡아냅니다.

{% hint style="info" %}
**하이브리드 검색**은 키워드 검색과 의미 기반 검색의 결과에 각각 점수를 매긴 뒤, 두 점수를 합산하여 최종 순위를 정합니다. 하나의 방식만 쓸 때보다 더 정확한 결과를 얻을 수 있어서, 실제 서비스에서 가장 널리 사용되는 방식입니다.
{% endhint %}

***

### Graph RAG : 관계까지 이해하는 검색

RAG는 질문과 관련된 문서를 찾아 AI에게 전달합니다.\
하지만 한 가지 약점이 있습니다. 정보가 **여러 문서에 흩어져** 있으면 제대로 답하기 어렵습니다.

예를 들어 봅시다.

> 질문: "김 팀장이 담당하는 프로젝트의 고객사는 어디인가요?"

이 질문에 답하려면 세 가지 정보가 필요합니다.

1. "김 팀장"이 누구인지 (인사 문서)
2. 김 팀장이 어떤 "프로젝트"를 담당하는지 (업무 배정 문서)
3. 그 프로젝트의 "고객사"가 어디인지 (계약 문서)

일반 RAG는 질문과 가장 비슷한 문서 몇 개만 검색합니다.\
"김 팀장" 관련 문서는 찾겠지만, 거기서 프로젝트를 확인하고, 다시 그 프로젝트의 고객사를 찾아가는 **연쇄적인 탐색**은 하지 못합니다.

**Graph RAG**는 이 문제를 해결합니다.\
문서의 내용을 미리 분석하여 **정보 간의 관계를 지도(그래프)**&#xB85C; 만들어 둡니다.

```
김 팀장 ──[담당]──→ 스마트팩토리 프로젝트 ──[고객사]──→ A업체
```

이 지도가 있으면, "김 팀장"에서 출발하여 "담당 프로젝트"를 거쳐 "고객사"까지 **관계를 따라가며** 답을 찾을 수 있습니다.

| 비교    | 일반 RAG           | Graph RAG       |
| ----- | ---------------- | --------------- |
| 검색 방식 | 질문과 비슷한 문서를 찾음   | 정보 간의 관계를 따라감   |
| 비유    | 도서관에서 관련 책을 찾는 것 | 지도에서 경로를 따라가는 것 |
| 강점    | 단일 문서 안의 답변      | 여러 문서에 걸친 복합 질문 |

{% hint style="info" %}
**그래프(Graph)**&#xB294; 점(정보)과 선(관계)으로 이루어진 구조입니다. 지하철 노선도를 떠올리면 됩니다. 역(정보)과 노선(관계)이 연결되어 있어서, 한 역에서 다른 역까지 어떻게 갈 수 있는지를 알 수 있습니다.
{% endhint %}

### 에이전틱 서치 : AI가 직접 찾아다니는 검색

지금까지 살펴본 방식들은 모두 **사람이 질문하면 시스템이 한 번 검색하고 답하는** 구조입니다.\
하지만 복잡한 질문에는 한 번의 검색으로 충분하지 않을 때가 있습니다.

> 질문: "지난 분기 매출이 가장 높은 제품의 담당자 연락처를 알려줘"

이 질문에 답하려면 사람이라면 이렇게 할 것입니다.

1. 먼저 지난 분기 매출 보고서를 찾는다
2. 매출 1위 제품이 무엇인지 확인한다
3. 그 제품의 담당자가 누구인지 찾는다
4. 담당자의 연락처를 찾는다

**에이전틱 서치(Agentic Search)**&#xB294; AI가 이 과정을 **스스로 수행**합니다.\
한 번 검색하고 끝나는 것이 아니라, 검색 결과를 보고 **다음에 무엇을 더 찾아야 하는지 판단**하여 여러 번 검색을 이어갑니다.

<table><thead><tr><th width="155.88671875">비교</th><th>기존 검색 (RAG)</th><th>에이전틱 서치</th></tr></thead><tbody><tr><td>검색 횟수</td><td>1회</td><td>필요한 만큼 여러 회</td></tr><tr><td>판단 주체</td><td>사람이 질문, 시스템이 검색</td><td><strong>AI가 스스로 판단</strong>하여 추가 검색</td></tr></tbody></table>

실제 과정은 이렇습니다.

1. AI가 질문을 분석합니다: "매출 → 제품 → 담당자 → 연락처, 순서대로 찾아야겠다"
2. 먼저 매출 보고서를 검색합니다 → "A제품이 1위"라는 정보를 얻습니다
3. 이 결과를 바탕으로 A제품 담당자를 검색합니다 → "홍길동"이라는 정보를 얻습니다
4. 홍길동의 연락처를 검색합니다 → 최종 답변을 생성합니다

AI 코딩 도구인 Claude Code가 이 방식의 좋은 예입니다.\
개발자가 "이 버그를 고쳐 줘"라고 말하면, AI가 직접 프로젝트의 파일들을 탐색하고, 관련 코드를 찾아 읽고, 어디를 수정해야 하는지 스스로 판단합니다.

{% hint style="info" %}
**에이전트(Agent)**&#xB780; 스스로 판단하고 행동하는 AI를 말합니다. 에이전틱 서치는 이 에이전트가 "검색"이라는 행동을 스스로 계획하고 반복하는 것입니다. 마치 유능한 비서가 "이것 좀 알아봐 줘"라는 한마디에 여러 곳을 돌아다니며 정보를 모아 정리해 주는 것과 같습니다.
{% endhint %}

### 한 줄 정리

> 검색은 **키워드 매칭**에서 **의미 이해**로, **하이브리드 검색**으로 두 장점을 합치고, **RAG**로 AI가 답변을 생성하며, **Graph RAG**로 관계를 따라가고, **에이전틱 서치**로 AI가 스스로 찾아다니는 단계까지 진화하고 있습니다.

여기까지, IT 서비스의 전체 구조를 함께 살펴보았습니다.

> **화면**에서 사용자가 행동하면, **네트워크**를 통해 **서버**로 요청이 전달되고, **데이터베이스**에서 데이터를 가져와 처리한 뒤, 다시 화면에 결과가 표시됩니다.\
> 이 과정에서 데이터는 **JSON** 같은 형식으로 포장되어 오가고, 모든 프로그램은 **프로그래밍 언어**로 만들어져 **운영체제** 위에서 실행됩니다.\
> 그리고 최근에는 **검색과 AI의 결합**으로, 단순히 정보를 찾는 것을 넘어 AI가 스스로 판단하고 답을 만들어주는 시대가 열리고 있습니다.

이 기본 구조를 이해하고 나면, 앞으로 어떤 IT 서비스를 만나더라도 "아, 이것은 이런 구조로 동작하고 있구나"라고 전체 그림을 그릴 수 있을 것입니다.
