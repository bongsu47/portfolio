# 의도분류

의도 분류 노드는 사용자의 질문이 어떤 의도인지 **AI가 판단하여**, 적절한 경로로 안내하는 노드입니다.\
하나의 앱에서 여러 종류의 질문을 처리해야 할 때, 질문의 유형에 따라 서로 다른 흐름으로 분기시킬 수 있습니다.

***

## 어떤 역할을 하나요?

의도 분류 노드는 사용자의 질문을 분석하여 미리 정의한 의도 중 하나로 분류합니다.\
분류 결과에 따라 워크플로우를 서로 다른 방향으로 분기시킬 수 있습니다.

예를 들어, 사내 인사 챗봇을 만든다고 가정하면 다음과 같이 동작합니다.

* "연차 며칠 남았어?" → **인사 관련 질의**로 분류 → 지식 검색 + LLM으로 답변
* "모니터 고장났는데 어디에 말하면 돼?" → **문의 사항 접수**로 분류 → 담당 부서에 메일 발송

각 의도별로 서로 다른 노드를 연결하면, 질문 유형에 맞는 처리를 할 수 있습니다.

> 시작 → **의도 분류** → 인사 관련 질의 → 지식 검색 → LLM → 답변\
> 　　　　　　　　└→ 문의 사항 접수 → LLM (접수 내용 정리) → 메일 발송 도구 → 답변

***

## 조건 노드와 무엇이 다른가요?

의도 분류 노드와 조건 노드는 둘 다 워크플로우를 분기시키는 노드이지만, **분기 기준이 근본적으로 다릅니다.**

**조건 노드**는 변수의 값을 규칙으로 비교합니다.\
"포함", "같음", "보다 큼" 같은 명확한 조건을 직접 설정합니다.\
예: `status`가 `active`이면 A로, 아니면 B로.

**의도 분류 노드**는 AI가 텍스트의 의미를 읽고 판단합니다.\
같은 뜻이라도 다양하게 표현되는 자연어를 이해하여 분류합니다.\
예: "연차 신청 어떻게 해?", "휴가 쓰고 싶은데", "남은 연차가 궁금해" → 모두 **인사 관련 질의**로 분류.

|        | 의도 분류              | 조건 노드                     |
| ------ | ------------------ | ------------------------- |
| 분기 기준  | AI가 텍스트의 의미를 판단    | 값을 규칙으로 비교 (포함, 같음, 초과 등) |
| 비용     | AI 모델을 호출하므로 비용 발생 | 비용 없음 (단순 연산)             |
| 속도     | 모델 호출 시간 필요        | 즉시 처리                     |
| 분기 수   | 의도 개수만큼 자유롭게       | 조건별로 설정                   |
| 적합한 상황 | "이 질문이 어떤 유형인지" 파악 | "이 값이 무엇인지" 확인            |

자연어 텍스트의 의도를 파악해야 한다면 의도 분류를, 명확한 값 비교가 필요하다면 조건 노드를 사용하세요.\
조건 노드로 충분한 상황이라면 조건 노드를 쓰는 것이 비용과 속도 면에서 유리합니다.

***

## 설정 방법

의도 분류 노드를 클릭하면 설정 패널이 열립니다.<br>

<figure><img src="../../../.gitbook/assets/image (662).png" alt="" width="328"><figcaption></figcaption></figure>

### 1. 모델

의도를 판단할 AI 모델을 선택합니다.\
모델 이름 옆의 설정 아이콘을 클릭하면 Temperature 등 모델 파라미터를 조절할 수 있습니다.

의도 분류는 비교적 간단한 작업이므로, 가벼운 모델로도 충분한 경우가 많습니다.\
다만 의도의 수가 많거나 구분이 모호한 경우에는 고성능 모델이 더 정확합니다.

### 2. 입력 변수 (필수 \*)

분류할 텍스트가 담긴 변수를 선택합니다.\
클릭하면 선택 가능한 변수 목록이 나타납니다.

가장 일반적인 선택은 `sys.query` (사용자가 입력한 원래 질문)입니다.\
이전 LLM 노드가 가공한 텍스트를 선택할 수도 있습니다.

### 3. 비전

토글을 켜면 **이미지를 함께 분석**하여 의도를 판단합니다.\
비전을 지원하는 모델에서만 사용 가능합니다.

예를 들어, 사용자가 고장난 장비 사진을 보내며 "이거요"라고만 입력해도, 이미지를 분석하여 "문의 사항 접수"로 분류할 수 있습니다.

### 4. 의도

분류할 의도(카테고리)를 정의합니다.\
기본으로 **의도1**, **의도2** 두 개가 있으며, 오른쪽 상단의 `+` 버튼으로 더 추가할 수 있습니다.

각 의도의 텍스트 영역에는 **해당 의도에 어떤 질문이 해당하는지** 설명을 작성합니다.\
AI는 이 설명을 읽고, 입력 텍스트가 어떤 의도에 가장 맞는지 판단합니다.

예를 들어 사내 인사 챗봇이라면 다음과 같이 정의합니다.

| 의도  | 설명 예시                                             |
| --- | ------------------------------------------------- |
| 의도1 | 연차, 휴가, 급여, 복리후생, 근태, 인사 제도 관련 질문                 |
| 의도2 | 시설 고장, 장비 요청, 업무 불편 사항, 건의 사항 등 담당 부서에 전달이 필요한 문의 |
| 의도3 | 위에 해당하지 않는 일반 대화, 인사, 잡담                          |

의도를 추가하면 노드에 해당 의도별 **연결 포인트**가 생깁니다.\
각 연결 포인트에 서로 다른 다음 노드를 연결하면, 의도별로 다른 흐름을 만들 수 있습니다.

* **의도1** (인사 관련 질의) → 지식 검색 노드 → LLM 노드 → 답변\
  인사 규정 지식에서 관련 내용을 검색하고, LLM이 답변을 생성합니다.
* **의도2** (문의 사항 접수) → LLM 노드 (접수 내용 정리) → 메일 발송 도구 → 답변\
  LLM이 문의 내용을 정리한 뒤, 담당 부서에 메일을 발송하고 접수 완료를 안내합니다.
* **의도3** (기타) → LLM 노드 → 답변\
  간단한 인사나 잡담에 대응합니다.

{% hint style="info" %}
#### 의도를 잘 정의하려면?

* **구체적으로 작성하세요**: "기타"보다는 "일반 대화 및 인사"처럼 구체적으로 작성하면 AI가 더 정확하게 분류합니다.
* **의도 수를 적정하게 유지하세요**: 의도가 너무 많으면(10개 이상) 분류 정확도가 떨어질 수 있습니다. 비슷한 의도는 하나로 합치는 것을 권장합니다.
* **겹치지 않게 정의하세요**: "인사 질문"과 "HR 문의"처럼 의미가 비슷한 의도가 있으면 AI가 혼란스러워합니다. 각 의도의 경계가 명확해야 합니다.
{% endhint %}

### 5. 프롬프트 (지시사항)

AI가 분류할 때 참고할 추가 지침을 작성할 수 있습니다.\
`/`를 입력하여 변수를 삽입할 수도 있습니다.

의도 정의만으로 구분이 모호한 경우, 여기에 구체적인 규칙을 적으면 분류 정확도가 높아집니다.\
지시사항을 비워두면 AI가 의도 정의만 보고 판단합니다. 간단한 분류라면 비워둬도 충분합니다.

아래는 인사 챗봇에서 사용할 수 있는 지시사항 작성 예시입니다.

```
## 분류 원칙
- 사용자의 핵심 의도를 기준으로 가장 적합한 의도 1개만 선택한다.
- 두 가지 의도가 섞여 있으면, 더 구체적인 의도를 우선한다.
- 근거가 부족하거나 판단이 어려우면 "기타"를 선택한다.

## 경계 규칙
- 인사 제도/정책에 대한 질문 → "인사 관련 질의"
- 본인의 연차/급여/근태 조회 → "인사 관련 질의"
- 시설/장비 고장 신고 → "문의 사항 접수"
- 업무 환경 개선 건의 → "문의 사항 접수"
- 특정 부서/담당자 연결 요청 → "문의 사항 접수"
- 인사 규정 변경 건의 → "문의 사항 접수" (질의가 아닌 접수)
- 인사, 감사, 잡담 → "기타"
```

이처럼 의도 간 경계가 모호한 케이스를 구체적으로 명시하면 분류 정확도가 높아집니다.\
예를 들어 "인사 규정 변경 건의"는 인사 관련이지만 질의가 아니라 접수이므로, 이런 경계를 명확히 해두면 오분류를 줄일 수 있습니다.

### 6. 메모리

토글을 켜면 **이전 대화 내용을 함께 고려**하여 분류합니다.

예를 들어, 이전 대화에서 "연차"에 대해 이야기하고 있었다면, 다음 메시지가 "그럼 어떻게 하면 돼?"처럼 모호하더라도 "인사 관련 질의"로 정확하게 분류할 수 있습니다.

대화형 워크플로우(챗플로우)에서 문맥이 중요한 경우 켜는 것을 권장합니다.

***

## 출력 변수

| 변수           | 타입     | 설명         |
| ------------ | ------ | ---------- |
| `class_name` | String | 분류된 의도의 이름 |

출력 변수 외에도, 분류 결과에 따라 **노드 자체가 해당 의도의 연결 포인트로 분기**합니다.\
`class_name`은 이후 노드에서 "어떤 의도로 분류되었는지" 텍스트로 참조할 때 사용합니다.
