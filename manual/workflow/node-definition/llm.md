# LLM

LLM 모델을 호출하여 텍스트를 생성하는 노드입니다.

사용자의 질문에 답변하거나, 문서를 요약하거나, 번역하는 등 워크플로우에서 AI가 필요한 거의 모든 작업에 사용됩니다. 워크플로우의 핵심 노드라고 할 수 있습니다.

<figure><img src="../../../.gitbook/assets/image (37).png" alt="" width="286"><figcaption></figcaption></figure>

***

## 어떤 역할을 하나요?

LLM 노드는 입력된 텍스트를 이해하고, 프롬프트(지시사항)에 따라 새로운 텍스트를 생성합니다.

예를 들어,

* 사용자의 질문에 대해 자연스러운 답변을 생성할 수 있습니다
* 긴 문서를 요약하거나, 다른 언어로 번역할 수 있습니다
* 지식 검색 결과를 바탕으로 근거 있는 답변을 만들 수 있습니다
* 텍스트를 분석하여 감정, 카테고리, 핵심 키워드 등을 추출할 수 있습니다

생성된 텍스트는 이후 답변 노드, 조건 노드 등 다양한 노드에서 활용할 수 있습니다.

***

## 설정 방법

### 1. 모델 선택

사용할 AI 모델을 선택합니다. 모델에 따라 성능, 속도, 비용이 달라지므로 용도에 맞는 모델을 선택하는 것이 중요합니다.

일반적으로 많이 사용되는 모델의 구분은 아래와 같습니다.

| 작업 유형                                       | 추천 모델                                                     | 특징                    |
| ------------------------------------------- | --------------------------------------------------------- | --------------------- |
| <p>간단한 작업<br>(분류, 짧은 답변, 포맷 변환 등)</p>       | <p>Claude Haiku<br>GPT Nano, GPT Mini<br>Gemini Flash</p> | 빠르고 가벼움비용 효율적         |
| <p>복잡한 작업<br>(긴 문서 요약, 추론, 일반 코딩 등)</p>     | <p>Claude Sonnet<br>GPT<br>Gemini Pro</p>                 | 정확도와 성능 균형안정적인 결과     |
| <p>고난도 작업<br>(정교한 추론, 어려운 코딩, 전략적 사고 등)</p> | <p>Claude Opus<br>GPT Pro</p>                             | 최고 수준의 성능깊은 이해와 추론 가능 |

정답은 없으며, 같은 프롬프트라도 모델에 따라 결과가 달라질 수 있습니다. 여러 모델을 테스트해보고 목적에 맞는 최적의 모델을 선택하는 것을 권장합니다.

{% hint style="info" %}
#### 모델을 구분하는 방법

모델은 보통 **제조사-시리즈-버전** 구조로 제공됩니다.&#x20;

예를 들어 Claude, GPT, Gemini와 같은 제조사별 모델이 있으며, 각 모델은 Haiku, Sonnet, Opus처럼 성능 단계별 시리즈로 나뉘고, 그 뒤에 4.5와 같은 숫자 형태의 버전이 붙습니다.

같은 시리즈 내에서는 최신 버전과 이전 버전의 요금이 동일한 경우가 많기 때문에, 특별한 이유가 없다면 최신 버전을 사용하는 것을 권장합니다.

**단, 모델이나 버전이 변경되면 기존에 작성한 프롬프트의 결과가 이전과 다르게 나올 수 있습니다. 따라서 중요한 워크플로우에서는 모델 변경 시 반드시 결과를 다시 확인하는 것이 좋습니다**.
{% endhint %}

{% hint style="info" %}
#### Antropic Claude 와 Bedrock Claude가 다른건가요?

그냥 둘은 같은 Claude AI지만, **사용 환경이 다릅니다.**

* **Anthropic Claude**: 일반 서비스처럼 바로 사용하는 AI
* **Bedrock Claude**: AWS 안에서 기업용으로 사용하는 AI
  * **Bedrock은 입력한 데이터가 학습에 쓰이지 않고, 보안 관리가 가능한 환경에서 사용됩니다.**
{% endhint %}

{% hint style="info" %}
#### 항상 미소에서 지원하는 최신 모델이 자동 적용되게 할 수 있나요?

**Latest 모델**을 선택하면 가능합니다.

아래와 같이 모델 공급사의 모델 Family 별로 "Latest" 표기가된 모델을 리스트에서 선택할 수 있습니다.

해당 모델을 선택할 경우 현재 미소 플랫폼에서 공식 지원하고있으며, 해당 워크스페이스에서 사용 가능한 최신 모델이 자동 적용됩니다.

따라서 사용자는 모델 버전 업데이트시 추가적인 설정 작업 없이 항상 최신 모델을 내 앱에 반영할 수 있습니다.

![](<../../../.gitbook/assets/image (676).png>)
{% endhint %}

### 2. 모델 설정

AI의 답변 스타일을 세밀하게 조절할 수 있는 고급 설정입니다. 모델에 따라 표시되는 항목이 다를 수 있으며, 각 항목은 토글 스위치로 활성화/비활성화할 수 있습니다.

<figure><img src="../../../.gitbook/assets/image (618).png" alt="" width="394"><figcaption></figcaption></figure>

LLM은 글을 쓸 때, 매 순간 "다음에 올 단어 후보"에 확률을 매깁니다. 예를 들어 어떤 순간에 이런 후보가 있을 수 있습니다.

{% code expandable="true" %}
```
감사합니다: 40%
좋습니다: 25%
가능합니다: 15%
흥미롭습니다: 8%
독특합니다: 5%
그 외 표현들: 나머지
```
{% endcode %}

Temperature, Top P, Top K는 각각 다른 방식으로 이 후보 목록을 조절하는 설정입니다.

<figure><img src="../../../.gitbook/assets/image (619).png" alt="" width="563"><figcaption></figcaption></figure>

#### **1) Temperature**

AI가 가장 확률 높은 단어를 고집할지, 아니면 다른 표현도 자유롭게 시도할지를 결정합니다. 후보 단어들의 확률 차이를 더 벌리거나 줄이는 역할입니다.

| 항목  | 내용         |
| --- | ---------- |
| 범위  | 0.0 \~ 1.0 |
| 기본값 | 0.0        |

* **낮을수록**: 가장 확률 높은 단어를 강하게 선택합니다. 답변이 안정적이고 일관적이지만, 표현이 비슷비슷해질 수 있습니다.
* **높을수록**: 확률이 조금 낮은 단어도 자주 선택합니다. 표현이 다양해지지만, 가끔 예상 밖 답변이 나올 수 있습니다.

쉽게 말하면, "얼마나 모범답안 위주로 갈지"를 정하는 값입니다. 낮으면  가장 무난한 답만 고르고, 높으면 좀 더 자유롭게 여러 표현을 시도합니다.

* **정확성이 중요할 때** (FAQ, 매뉴얼 안내, 데이터 추출): 0.0\~0.3
* **창의성이 필요할 때** (아이디어, 마케팅 문구, 글쓰기): 0.7\~1.0

#### **2) Top P**

후보 단어를 확률 높은 순서대로 나열한 뒤, 누적 확률이 설정한 값에 도달할 때까지의 후보만 남기고 나머지는 버립니다.

| 항목  | 내용         |
| --- | ---------- |
| 범위  | 0.0 \~ 1.0 |
| 기본값 | 1.0        |

예를 들어 Top P를 0.8로 설정하면, 확률이 높은 단어부터 차례로 더해서 총합이 80%가 될 때까지만 후보로 사용합니다. 그보다 확률이 낮은 단어들은 제외됩니다.

쉽게 말하면, 지원자들을 점수순으로 세운 뒤 “합격권 안에 드는 사람들만 남기고 나머지는 제외하는” 방식입니다. 너무 점수가 낮은 지원자는 미리 걸러내고, 어느 정도 가능성 있는 상위 지원자들 중에서만 최종 선택하는 것과 비슷합니다.

* 낮을수록 : 선택지가 줄어들어 답변이 더 안정적이고 예측 가능해집니다​
* 1.0 (기본값): 모든 후보를 그대로 사용합니다.

#### **3) Top K**

확률 높은 순서대로 상위 K개의 단어만 남기고, 나머지는 전부 제외합니다.

| 항목  | 내용       |
| --- | -------- |
| 범위  | 1 \~ 100 |
| 기본값 | 50       |

Top P가 "비율"로 후보를 거르는 것이라면, Top K는 "개수"로 후보를 거르는 것입니다. 지원자 중 무조건 상위 50명만 면접 보는 방식이라고 생각하면 됩니다.

* **작을수록**: 후보가 적어 안정적이지만 단조로울 수 있습니다.
* **클수록**: 후보가 많아져 다양성이 늘어납니다.

{% hint style="info" %}
셋 다 AI의 단어 선택에 영향을 주지만, Temperature는 확률 분포의 **모양**을 바꾸고, Top P와 Top K는 후보 자체를 **잘라내는** 역할입니다. 대부분의 경우 **Temperature만 조절하면 충분**하며, Top P와 Top K는 기본값 그대로 두어도 좋습니다.
{% endhint %}

#### **4) Presence Penalty (주제 반복 방지)**

이미 언급된 내용이 다시 등장하는 것을 억제합니다.

| 항목  | 내용         |
| --- | ---------- |
| 범위  | 0.0 \~ 1.0 |
| 기본값 | 0.0        |

값을 높이면 AI가 같은 주제나 단어를 반복하지 않고, 새로운 내용을 다루려고 합니다. 길고 다양한 답변이 필요할 때 유용합니다.

#### **5) Frequency Penalty (표현 반복 방지)**

자주 사용된 단어가 또 사용되는 것을 억제합니다.

| 항목  | 내용         |
| --- | ---------- |
| 범위  | 0.0 \~ 1.0 |
| 기본값 | 0.0        |

Presence Penalty와 비슷하지만, 이 설정은 같은 단어가 **여러 번** 반복될수록 더 강하게 억제합니다. 예를 들어, "매우 매우 매우 좋습니다" 같은 반복을 줄이고 싶을 때 효과적입니다.

#### **6) Max Tokens (최대 답변 길이)**

LLM이 생성할 답변의 최대 길이를 토큰 단위로 제한합니다.

| 항목  | 내용     |
| --- | ------ |
| 범위  | 모델별 상이 |
| 기본값 | 모델별 상이 |

답변이 중간에 잘리는 경우, 이 값을 늘려보세요. 반대로, 짧은 답변만 필요한 경우(예: 분류, 예/아니오 판정) 값을 줄이면 불필요한 토큰 소모를 방지할 수 있습니다.

#### **7) Response Format (응답 형식)**

LLM의 응답이 특정 형식을 따르도록 강제합니다.&#x20;

특히 프롬프트에서 응답 예시를 구체적으로 함께 제시하면, 모델이 원하는 형식을 더 정확하게 따르도록 유도할 수 있습니다.

| 옵션   | 설명                  |
| ---- | ------------------- |
| JSON | 응답을 JSON 형식으로 출력합니다 |
| XML  | 응답을 XML 형식으로 출력합니다  |

{% hint style="info" %}
#### 파라미터, 어떻게 설정해야 하나요?

대부분의 경우 **Temperature만 조절**하면 충분합니다. 나머지는 기본값 그대로 두어도 좋습니다.

* **정확한 답변이 중요하다면** (FAQ, 지식 기반 답변, 데이터 처리): Temperature를 0에 가깝게 설정하세요.
* **다양한 표현이 필요하다면** (글쓰기, 아이디어, 마케팅 문구): Temperature를 0.7\~1.0으로 올리세요.
* **답변이 같은 말을 반복한다면**: Presence Penalty나 Frequency Penalty를 0.3\~0.5 정도로 설정해보세요.
* **답변이 중간에 잘린다면**: Max Tokens 값을 늘려보세요.

Temperature와 Top P를 동시에 변경하는 것은 권장하지 않습니다. 둘 다 답변의 다양성에 영향을 주기 때문에, 의도치 않은 결과가 나올 수 있습니다.
{% endhint %}

### 3. 프롬프트 작성

프롬프트는 AI에게 전달하는 지시사항입니다. 프롬프트의 품질이 AI 답변의 품질을 결정하므로, 가장 중요한 설정입니다.

프롬프트를 잘 작성하는 방법은 아래 문서를 참고바랍니다.

{% content-ref url="../../../study/prompt/" %}
[prompt](../../../study/prompt/)
{% endcontent-ref %}

프롬프트의 역할별로 메시지를 구분하여 작성할 수 있습니다.

<figure><img src="../../../.gitbook/assets/image (621).png" alt="" width="216"><figcaption></figcaption></figure>

#### **1) 시스템 메시지**

AI의 역할과 규칙을 정의합니다. 워크플로우가 실행되는 동안 항상 유지되는 기본 지침입니다.

예시:

```
<role>
당신은 어떤 상황에도 침착하게 대응하는 성격을 가진 MISO 고객센터의 AI 상담원입니다.
<role>

<rules>
- 항상 존댓말을 사용하세요
- 모르는 질문에는 "담당자에게 연결해드리겠습니다"라고 답변하세요
- 제공된 참고 자료에 근거하여 답변하세요
</rules>
```

#### **2) 사용자 메시지**

AI에게 전달할 질문이나 요청을 작성합니다. 변수를 삽입하여 동적으로 내용을 구성할 수 있습니다.

예시:

```
다음 질문에 답변해주세요.

질문: {{sys.query}}
```

#### **3) 어시스턴트 메시지**

AI가 어떤 식으로 응답해야 하는지 예시를 미리 보여주는 메시지입니다. 시스템 메시지가 "규칙"을 알려주는 것이라면, 어시스턴트 메시지는 "이런 식으로 답변하면 돼"라는 **모범 답안**을 보여주는 역할입니다.

```
<output_example>
질문 : A 상품 반품해줘

답변:
네, 고객님. 해당 제품의 반품 절차를 안내드리겠습니다.
​
1. 마이페이지 > 주문내역에서 반품 신청을 눌러주세요.
2. 반품 사유를 선택해주세요.
3. 택배 수거일을 지정해주세요.

추가 문의사항이 있으시면 말씀해주세요.
</output_example>
```

위 예시를 어시스턴트 메시지로 넣어두면, AI는 이 형식과 말투를 참고하여 이후 답변을 비슷한 스타일로 생성합니다.

{% hint style="info" %}
#### 메시지를 꼭 나눠서 입력해야 하나요?

꼭 나눌 필요는 없습니다. 사용자 메시지 하나에 역할 설명, 지시사항, 질문을 모두 넣어도 동작합니다.

하지만 역할별로 나누면 몇 가지 이점이 있습니다.

* **시스템 메시지**는 AI가 "기본 규칙"으로 인식하여, 이후 어떤 질문이 와도 일관되게 지킵니다.
* **사용자 메시지 → 어시스턴트 메시지** 쌍을 여러 개 넣으면, AI가 "이런 질문에는 이렇게 답하면 되는구나"라는 패턴을 학습합니다. (Few-shot 학습)
* 메시지를 분리하면 프롬프트 관리와 수정이 더 편합니다.

간단한 작업이라면 시스템 메시지 + 사용자 메시지만으로 충분하고, 답변을 일정하게 출력하고 싶을때 어시스턴트 메시지를 추가하세요.
{% endhint %}

{% hint style="info" %}
#### 일부 모델은 사용자 메시지가 필수입니다

OpenAI의 o1, o3 시리즈 등 일부 모델은 시스템 메시지만으로는 동작하지 않으며, 반드시 사용자 메시지가 포함되어야 합니다. 이 경우 사용자 메시지 없이 실행하면 오류가 발생할 수 있습니다.

안정적인 사용을 위해, 프롬프트 구성 시 **사용자 메시지는 항상 포함하는 것을 권장**합니다.
{% endhint %}

***

### 4.  배경지식 (컨텍스트)&#x20;

지식 검색 노드의 결과를 AI에게 배경 지식으로 전달하기 위한 설정입니다.

중요한 점은, AI에게 전달되는 모든 정보는 **프롬프트 영역에 포함되어야** 한다는 것입니다. 컨텍스트는 지식 검색 결과를 프롬프트에 연결하는 수단일 뿐, 컨텍스트에 변수를 할당했다고 자동으로 AI에게 전달되지 않습니다. 아래 두 단계를 모두 완료해야 합니다.

**1) 배경지식 영역에 변수 할당**

LLM 노드의 배경지식 설정에서, 지식 검색 노드의 결과 변수를 선택합니다. 이 단계는 "어떤 데이터를 배경 지식으로 쓸지" 지정하는 것입니다.

**2) 프롬프트에 배경지식 블록 삽입**

배경지식 영역에 변수를 할당하면, 프롬프트 편집기에 배경지식 블록을 삽입할 수 있는 버튼이 나타납니다. 이 블록을 프롬프트 안에서 원하는 위치에 삽입해야, 해당 위치에 지식 검색 결과가 들어갑니다.

예를 들어, 시스템 메시지에 배경지식 블록을 삽입하면 이런 형태가 됩니다.

````
아래 참고 자료를 근거로 답변해주세요.
​```
{{배경지식 변수를 '/' 로 할당}}
​```
참고 자료에 없는 내용은 "확인이 어렵습니다"라고 답변하세요.
````

<figure><img src="../../../.gitbook/assets/image (620).png" alt="" width="397"><figcaption></figcaption></figure>

{% hint style="info" %}
#### 배경지식 블록 없이 프롬프트에 변수로 직접 넣어도 되나요?

* 프롬프트 안에서 `/`를 눌러 지식 검색 결과 변수를 직접 삽입하는 것도 가능하며, AI에게 전달되는 텍스트 내용은 동일합니다.
* **컨텍스트를 통해 연결하면 "인용 및 출처" 기능이 동작합니다.**&#x20;
  * 지식 검색 결과에는 텍스트 내용뿐 아니라 부가 정보(어떤 지식에서 가져왔는지, 문서 이름, 페이지, 관련도 점수 등)가 포함되어 있는데, 컨텍스트를 사용하면 이 부가 정보가 시스템에 자동으로 기록됩니다.
  * 앱 설정에서 "인용 및 출처" 기능을 켜면, 사용자가 AI 답변 아래에서 참고된 문서 목록을 확인하고 원본을 다운로드할 수 있습니다. 변수로 직접 넣으면 텍스트만 전달되므로 이 기능이 동작하지 않습니다.
{% endhint %}

***

#### 4. 메모리 (대화 기록)

이전 대화 내용을 AI에게 함께 전달하여, 대화의 맥락을 유지할 수 있습니다. 채팅 워크플로우에서만 사용 가능합니다.

<figure><img src="../../../.gitbook/assets/image (622).png" alt="" width="320"><figcaption></figcaption></figure>

| 설정        | 설명             | 기본값  |
| --------- | -------------- | ---- |
| 메모리 활성화   | 이전 대화를 포함할지 여부 | 비활성화 |
| 대화 윈도우 크기 | 포함할 이전 대화 쌍의 수 | 50   |

#### 메모리에는 어떤 내용이 저장되나요?

메모리에 저장되는 정보는 **사용자의 질문**과 **최종 답변** 두 가지뿐입니다.

* **사용자의 질문**: 사용자가 채팅창에 입력한 원래 메시지 (`sys.query`)
* **최종 답변**: 워크플로우의 답변 노드를 통해 사용자에게 전달된 최종 응답 텍스트

즉, 메모리는 "사용자가 무엇을 물었고, 앱이 무엇을 답했는지"를 한 쌍으로 기록합니다.

{% hint style="info" %}
#### 워크플로우에 LLM이 여러 개 있으면 어떻게 되나요?

워크플로우 내부에서 LLM 노드가 여러 개 직렬로 연결되어 있더라도, **메모리에 저장되는 것은 사용자의 원래 질문과 최종 답변뿐**입니다. 중간 LLM 노드들의 출력은 메모리에 저장되지 않습니다.

예를 들어, 아래와 같은 워크플로우가 있다고 가정합니다.

> 시작 → LLM 1 (요약) → LLM 2 (번역) → LLM 3 (답변 작성) → 답변

* LLM 1이 사용자의 질문을 요약한 결과 → 메모리에 저장되지 않음
* LLM 2가 번역한 결과 → 메모리에 저장되지 않음
* LLM 3이 작성한 내용 중 답변 노드를 통해 사용자에게 전달된 텍스트 → **메모리에 저장됨**

또한, 메모리를 활성화한 모든 LLM 노드는 **동일한 대화 기록**을 받습니다. LLM 1과 LLM 3 모두 메모리를 켜면, 둘 다 이전 라운드의 "사용자 질문 → 최종 답변" 기록을 동일하게 전달받습니다. 각 LLM의 중간 출력을 서로 기억하는 것이 아닙니다.
{% endhint %}

{% hint style="info" %}
#### 윈도우 크기는 어떻게 정하나요?

* **간단한 Q\&A 앱**: 5\~10이면 충분합니다. 이전 질문을 참고할 일이 적기 때문입니다.
* **상담/컨설팅 앱**: 20\~50이 적합합니다. 대화 흐름을 길게 기억해야 하기 때문입니다.
* **단발성 처리 앱**: 메모리를 비활성화하세요. 불필요한 토큰 소모를 줄일 수 있습니다.

윈도우 크기가 클수록 AI가 더 많은 맥락을 이해하지만, 그만큼 토큰 사용량이 늘어나고 비용이 증가합니다. 프롬프트가 모델의 컨텍스트 크기를 초과하면, 오래된 대화부터 자동으로 제거됩니다.
{% endhint %}

#### 5. 비전 (이미지 분석)

이미지를 AI에게 전달하여 분석할 수 있습니다. 비전을 지원하는 모델에서만 사용 가능합니다.

<figure><img src="../../../.gitbook/assets/image (623).png" alt="" width="323"><figcaption></figcaption></figure>

| 설정     | 설명                               |
| ------ | -------------------------------- |
| 비전 활성화 | 이미지 분석 기능 사용 여부                  |
| 파일 변수  | 분석할 이미지가 담긴 변수 선택                |
| 상세 수준  | 높음(정밀 분석, 토큰 많이 사용) 또는 낮음(빠른 분석) |

지원되는 파일 형식은 모델에 따라 다르지만, 일반적으로 PNG, JPG, GIF, WebP 이미지를 처리할 수 있습니다.

{% hint style="info" %}
비전 기능을 활성화했는데 비전을 지원하지 않는 모델을 선택하면, 이미지가 무시되고 텍스트만 처리됩니다. 모델 선택 시 비전 지원 여부를 확인하세요.
{% endhint %}

## 출력 변수

| 변수      | 타입     | 설명                         |
| ------- | ------ | -------------------------- |
| `text`  | String | AI가 생성한 텍스트 답변             |
| `usage` | Object | 토큰 사용량 정보 (입력/출력 토큰 수, 비용) |

LLM 노드의 모든 출력은 **`text` 하나의 변수에 문자열로 담겨서 나갑니다.** AI가 아무리 길고 복잡한 내용을 생성하더라도 결과는 항상 `text` 한 곳에 들어갑니다.

{% hint style="warning" %}
#### LLM 출력에서 여러 변수를 분리하고 싶다면?

LLM 노드 하나에서 여러 개의 변수를 나누어 내보내는 것은 불가능합니다. 출력은 항상 `text` 하나뿐입니다.

응답에서 특정 정보를 따로 뽑아 써야 한다면, 다음과 같은 방법을 사용합니다.

**1) 응답 형식을 JSON으로 설정 + 코드 노드 또는 템플릿 노드로 분리**

LLM 노드의 응답 형식을 JSON으로 설정하면 AI가 구조화된 JSON 텍스트를 출력합니다. 이 JSON 텍스트를 코드 노드나 템플릿 노드에서 파싱하여 원하는 항목을 개별 변수로 분리할 수 있습니다.

> 예: LLM에게 "제목, 요약, 키워드를 JSON으로 출력해줘"라고 지시 → `text`에 JSON 문자열 출력 → 코드 노드에서 제목, 요약, 키워드를 각각 별도 변수로 추출

**2) 변수 추출 노드 사용**

변수 추출 노드는 LLM을 활용하여 텍스트에서 원하는 정보를 자동으로 뽑아 여러 변수로 분리해 줍니다. 코드 작성 없이도 사용할 수 있어 간편합니다.

{% endhint %}
