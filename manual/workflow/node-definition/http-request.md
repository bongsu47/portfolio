# API 요청

API 요청 노드는 외부 서비스에 HTTP 요청을 보내고 응답을 받는 노드입니다.

<figure><img src="../../../.gitbook/assets/image (19).png" alt="" width="288"><figcaption></figcaption></figure>

다른 시스템과 데이터를 주고받거나, 외부 서비스의 기능을 활용할 때 사용합니다. 날씨 조회, 메일 발송, 사내 시스템 연동 등 워크플로우 안에서 처리할 수 없는 작업을 외부에 맡기는 창구 역할을 합니다.

***

## 어떤 역할을 하나요?

API 요청 노드는 외부 웹 서비스에 요청을 보내고, 돌아온 결과를 워크플로우에서 사용할 수 있게 합니다.

예를 들어,

* **공공 데이터 활용**: 기상청 API로 날씨 정보를 조회하거나, 공공 데이터 포털에서 인구 통계, 부동산 시세 등을 가져옵니다
* **외부 SaaS 연동**: Notion에 페이지를 생성하거나, Jira에 티켓을 등록하는 등 도구로 제공되지 않는 서비스를 직접 연결합니다

{% hint style="info" %}
#### 도구 vs API 요청, 어떤 걸 써야 하나요?

* **도구**: 이미 만들어진 연동 기능. 설정만 하면 바로 사용할 수 있어 간편합니다. 연동하려는 서비스가 도구 목록에 있다면 도구를 먼저 사용하세요.
* **API 요청 노드**: 도구에 없는 서비스를 직접 연동할 때 사용합니다. URL, 헤더, 본문 등을 직접 설정해야 하므로 API 문서를 참고해야 합니다.
{% endhint %}

{% hint style="info" %}
#### API가 무엇인가요?

API(Application Programming Interface)는 서로 다른 시스템이 데이터를 주고받기 위한 약속된 방식입니다. 택배를 보내려면 "주소, 받는 사람, 내용물"을 정해진 양식에 맞춰 작성하듯이, API도 정해진 형식에 맞춰 요청을 보내면 원하는 결과를 돌려받을 수 있습니다.

API를 사용하려면 해당 서비스의 API 문서를 참고하여 올바른 URL, 요청 형식 등을 확인해야 합니다. API 문서는 보통 해당 서비스의 개발자 페이지에서 제공됩니다.
{% endhint %}

## 설정 방법

<figure><img src="../../../.gitbook/assets/image (21).png" alt="" width="360"><figcaption></figcaption></figure>

### 1. API

가장 먼저 설정하는 항목입니다. **요청 방식**과 **URL**을 한 줄에 나란히 지정합니다.

#### **요청 방식 (HTTP 메서드)**

왼쪽 드롭다운에서 요청 방식을 선택합니다.

| 방식         | 용도        | 쉽게 말하면           |
| ---------- | --------- | ---------------- |
| **GET**    | 데이터 조회    | "이 정보 좀 알려줘"     |
| **POST**   | 데이터 전송/생성 | "이 데이터 좀 처리해줘"   |
| **PUT**    | 데이터 전체 수정 | "이 데이터를 통째로 바꿔줘" |
| **PATCH**  | 데이터 일부 수정 | "이 부분만 고쳐줘"      |
| **DELETE** | 데이터 삭제    | "이거 지워줘"         |
| **HEAD**   | 응답 헤더만 확인 | "내용 말고 상태만 알려줘"  |

대부분의 경우 **GET**(조회)과 **POST**(전송) 두 가지를 가장 많이 사용합니다. API 문서에 어떤 방식을 쓰라고 적혀 있으니 그대로 따르면 됩니다.

#### **URL**

오른쪽 입력란에 요청을 보낼 주소를 입력합니다.

```
https://api.example.com/weather
```

`/`를 입력하면 이전 노드의 변수를 선택할 수 있어, 주소의 일부를 동적으로 구성할 수도 있습니다.

```
https://api.example.com/orders/{{order_id}}
```

***

### 2. Header

요청에 추가 정보를 붙이는 영역입니다. "이 요청은 JSON 형식입니다", "인증 토큰은 이것입니다" 같은 부가 정보를 여기에 적습니다.

오른쪽 **+** 버튼을 눌러 항목을 추가하고, 왼쪽에 이름, 오른쪽에 값을 입력합니다.

| 이름                | 값                     | 설명                    |
| ----------------- | --------------------- | --------------------- |
| `Content-Type`    | `application/json`    | 보내는 데이터가 JSON 형식임을 알림 |
| `Authorization`   | `Bearer sk-abc123...` | API 인증 토큰 전달          |
| `Accept-Language` | `ko-KR`               | 한국어 응답 요청             |

{% hint style="info" %}
#### 인증이 필요한 API는 어떻게 하나요?

대부분의 외부 API는 인증 토큰(API 키)을 헤더에 포함해야 합니다. API 문서에 안내된 형식대로 헤더에 추가하면 됩니다.

예를 들어, API 문서에 "Bearer 토큰을 Authorization 헤더에 넣으세요"라고 되어 있다면:

* 이름: `Authorization`
* 값: `Bearer 여기에_API_키_입력`

API 키는 보안에 민감한 정보이므로, 실행 조건 설정의 환경 변수에 저장한 뒤 변수로 참조하는 것을 권장합니다.
{% endhint %}

각 입력란에서 `/`를 입력하면 변수를 선택할 수 있습니다. 필요 없는 항목은 오른쪽 휴지통 버튼으로 삭제합니다.

***

### 3. Params

URL 뒤에 붙는 검색 조건입니다. 주로 GET 요청에서 "어떤 데이터를 원하는지" 구체적으로 지정할 때 사용합니다.

오른쪽 **+** 버튼을 눌러 항목을 추가하고, 이름과 값을 입력합니다.

예를 들어 아래처럼 설정하면:

| 이름     | 값       |
| ------ | ------- |
| `city` | `Seoul` |
| `lang` | `kr`    |

실제 요청 URL은 `https://api.example.com/weather?city=Seoul&lang=kr`이 됩니다. URL 뒤에 `?`와 함께 자동으로 붙습니다.

***

### 4. Body

요청과 함께 보낼 데이터입니다. POST, PUT, PATCH 요청에서 주로 사용됩니다. GET 요청에서는 보통 비워둡니다.

오른쪽 드롭다운에서 데이터 형식을 선택합니다.

| 형식                        | 설명              | 언제 쓰나요?                       |
| ------------------------- | --------------- | ----------------------------- |
| **none**                  | 본문 없음           | GET 요청 등 데이터를 보낼 필요 없을 때      |
| **json**                  | JSON 형식         | 가장 일반적. 대부분의 API가 이 형식을 사용합니다 |
| **form-data**             | 폼 형식 (파일 첨부 가능) | 파일을 업로드할 때                    |
| **x-www-form-urlencoded** | URL 인코딩된 폼      | 간단한 폼 데이터를 보낼 때               |
| **raw-text**              | 일반 텍스트          | 특수한 형식이 필요할 때                 |

#### **JSON 본문 작성 예시**

형식을 **json**으로 선택하면 텍스트 편집기가 나타납니다. JSON 형식으로 데이터를 작성합니다.

```json
{
  "to": "user@example.com",
  "subject": "주문 완료 안내",
  "body": "주문이 정상적으로 접수되었습니다."
}
```

편집기 안에서 `/`를 입력하면 변수를 삽입할 수 있습니다. 예를 들어 사용자의 이메일을 동적으로 넣을 수 있습니다.

#### **form-data, x-www-form-urlencoded 작성 예시**

이름-값 쌍으로 입력합니다. **+** 버튼으로 항목을 추가합니다.

| 이름        | 값                  |
| --------- | ------------------ |
| `to`      | `user@example.com` |
| `subject` | `주문 완료 안내`         |

***

### 5. 타임아웃

외부 서비스의 응답을 얼마나 기다릴지 설정합니다. 슬라이더를 움직이거나 숫자를 직접 입력할 수 있습니다.

| 설정    | 설명               | 범위      |
| ----- | ---------------- | ------- |
| 연결 시간 | 서버와 연결이 될 때까지 대기 | 0\~300초 |
| 수신 시간 | 응답을 받을 때까지 대기    | 0\~120초 |
| 송신 시간 | 요청을 보낼 때까지 대기    | 0\~60초  |

{% hint style="info" %}
외부 API의 응답이 느려서 오류가 발생한다면 수신 시간을 늘려보세요. 반대로, 빠른 응답이 보장되는 API에서는 시간을 줄여 전체 워크플로우의 대기 시간을 단축할 수 있습니다.
{% endhint %}

## 출력 변수

API 요청이 완료되면 아래 4개의 변수가 출력됩니다.

| 변수            | 설명               |
| ------------- | ---------------- |
| `body`        | 응답 내용 (텍스트)      |
| `status_code` | 응답 상태 코드 (숫자)    |
| `headers`     | 응답 헤더 정보 (JSON)  |
| `files`       | 응답이 파일인 경우 파일 목록 |

## 주요 상태 코드

| 코드  | 의미      | 쉽게 말하면            |
| --- | ------- | ----------------- |
| 200 | 성공      | "요청 잘 처리했어"       |
| 400 | 잘못된 요청  | "보낸 내용에 문제가 있어"   |
| 401 | 인증 실패   | "API 키가 틀렸거나 없어"  |
| 403 | 권한 없음   | "이 요청을 할 권한이 없어"  |
| 404 | 찾을 수 없음 | "요청한 주소가 없어"      |
| 500 | 서버 오류   | "외부 서비스에 문제가 생겼어" |

{% hint style="warning" %}
#### status\_code를 꼭 확인하세요

API 요청 노드는 응답 상태 코드가 200이 아니더라도 오류로 중단되지 않습니다. 요청이 실패해도 워크플로우는 계속 진행됩니다.

따라서 API 요청 뒤에 **조건 노드**를 배치하여 `status_code`가 200인지 확인하는 것을 권장합니다. 200이면 정상 처리, 그 외에는 에러 안내 메시지를 보내는 흐름을 만들 수 있습니다.
{% endhint %}
