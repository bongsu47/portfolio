# 변수 할당

변수 할당 노드는 변수의 값을 변경하는 노드입니다.

<figure><img src="../../../.gitbook/assets/image (16).png" alt="" width="277"><figcaption></figcaption></figure>

기존 변수에 새로운 값을 넣거나, 계산을 통해 값을 업데이트할 수 있습니다. 주로 루프 노드와 함께 사용하여, 반복할 때마다 값을 업데이트하는 데 많이 활용됩니다.

***

## 어떤 역할을 하나요?

변수 할당 노드는 워크플로우 실행 중에 변수의 값을 직접 수정합니다. 다른 노드들이 "새로운 결과를 생성"하는 것과 달리, 이 노드는 "기존 변수의 값을 변경"합니다.

{% hint style="info" %}
#### 변수 할당의 대상

변수 할당 노드로 변경할 수 있는 변수는 **대화 변수**와 **루프 변수** 두 가지입니다. 일반 노드의 출력 변수는 변경할 수 없습니다.

* **대화 변수**: 채팅 워크플로우(챗플로우)에서 대화가 유지되는 동안 값이 보존되는 변수. 아래에서 자세히 설명합니다.
* **루프 변수**: 루프 노드에서 정의한 변수. 루프 안에서 값을 업데이트하는 데 사용합니다. 자세한 내용은 루프 문서를 참고하세요.
{% endhint %}

***

## 대화 변수란?

대화 변수는 **채팅 워크플로우(챗플로우) 전용** 기능으로, 대화가 진행되는 동안 값이 유지되는 변수입니다.

일반 워크플로우에서는 실행이 끝나면 모든 변수가 사라집니다. 하지만 챗플로우에서는 사용자가 여러 번 메시지를 주고받으므로, **이전 대화에서 저장한 값을 다음 대화에서도 사용**할 수 있어야 합니다. 이것이 대화 변수의 역할입니다.

예를 들어,

* 첫 번째 대화에서 사용자가 "내 이름은 홍길동이야"라고 말하면 → 대화 변수 `user_name`에 "홍길동" 저장
* 두 번째 대화에서 AI가 "홍길동님, 무엇을 도와드릴까요?"라고 응답 → 저장된 `user_name`을 활용

대화 변수가 없다면 AI는 매번 새로운 대화처럼 사용자의 이름을 다시 물어봐야 합니다.

***

### 대화 변수 추가하기

대화 변수는 워크플로우 편집 화면의 오른쪽 패널에서 추가합니다.

<figure><img src="../../../.gitbook/assets/image (17).png" alt="" width="392"><figcaption></figcaption></figure>

1. 오른쪽 패널에서 **앱 조건 설정**을 클릭합니다.
2. **대화 조건 설정** 탭을 선택합니다.
3. **+ 변수 추가** 버튼을 클릭합니다.
4. 변수 정보를 입력합니다.

| 항목  | 설명                            | 예시                              |
| --- | ----------------------------- | ------------------------------- |
| 이름  | 변수를 구분하는 식별자 (영문, 숫자, 밑줄만 가능) | `user_name`, `selected_product` |
| 유형  | 저장할 데이터의 종류                   | 문자열, 숫자, 배열 등                   |
| 기본값 | 대화 시작 시 초기값                   | `""`, `0`, `[]`                 |
| 설명  | 변수의 용도 메모 (선택사항)              | "사용자가 선택한 제품명"                  |

5. **저장** 버튼을 클릭하면 변수가 추가됩니다.

***

### 사용 가능한 데이터 유형

| 유형  | 설명                  | 기본값 예시  |
| --- | ------------------- | ------- |
| 문자열 | 텍스트                 | `""`    |
| 숫자  | 정수 또는 소수            | `0`     |
| 불리언 | 참/거짓                | `false` |
| 객체  | 여러 속성을 가진 데이터       | `{}`    |
| 배열  | 목록 (문자열, 숫자, 객체 배열) | `[]`    |

{% hint style="info" %}
#### 대화 조건 설정 vs 실행 조건 설정

앱 조건 설정에는 두 개의 탭이 있습니다.

* **대화 조건 설정**: 대화 중 정보를 기억하는 변수. 워크플로우 안에서 변수 할당 노드로 값을 변경할 수 있습니다.
* **실행 조건 설정**: API 키, 비밀번호 같은 민감한 정보를 안전하게 저장하는 변수. 앱을 내보내도 값이 포함되지 않습니다.

대화 중 값이 바뀌어야 하면 **대화 조건 설정**, 한 번 설정하면 바뀌지 않는 설정값이면 **실행 조건 설정**을 사용하세요.
{% endhint %}

***

### 대화 변수와 메모리는 다릅니다

챗플로우에는 **대화 변수** 외에 LLM 노드의 **메모리** 기능도 있습니다. 둘 다 "이전 내용을 기억한다"는 점에서 헷갈리기 쉽지만, 역할이 다릅니다.

**메모리**는 LLM 노드의 설정 중 하나로, 활성화하면 **이전 대화 기록(질문-답변 쌍)을 LLM에 함께 전달**합니다. 덕분에 LLM이 "아까 뭘 물어봤는지"를 알 수 있어 자연스러운 대화가 가능해집니다. 다만 메모리는 LLM이 대화 흐름을 참고하는 용도일 뿐, 워크플로우에서 꺼내 쓸 수 있는 변수가 아닙니다.

**대화 변수**는 워크플로우 제작자가 직접 만드는 변수입니다. 변수 할당 노드로 값을 저장하고, 조건 노드나 다른 LLM 노드에서 꺼내 쓸 수 있습니다.

|                | 대화 변수                            | 메모리 (LLM 노드 설정)           |
| -------------- | -------------------------------- | ------------------------- |
| **무엇인가요?**     | 제작자가 직접 만드는 변수                   | LLM 노드에 이전 대화 기록을 전달하는 기능 |
| **설정 위치**      | 앱 조건 설정 > 대화 조건 설정               | LLM 노드 > 메모리 활성화          |
| **기억하는 내용**    | 제작자가 지정한 특정 값 (예: 선택한 제품, 진행 단계) | 이전 대화의 질문-답변 쌍 전체         |
| **값 변경**       | 변수 할당 노드로 직접 변경                  | 자동 (대화가 쌓일수록 기록이 늘어남)     |
| **다른 노드에서 사용** | 가능 (조건 노드, 템플릿 노드 등에서 참조)        | 불가 (해당 LLM 노드만 참고)        |

{% hint style="info" %}
#### 언제 무엇을 쓸까요?

* "사용자가 아까 한 말을 LLM이 이해했으면 좋겠다" → **메모리** (LLM 노드에서 활성화)
* "사용자가 3단계 중 몇 단계까지 진행했는지 추적하고 싶다" → **대화 변수** (변수 할당 노드로 업데이트)
* 둘을 함께 사용할 수도 있습니다. 메모리로 대화 흐름을 유지하면서, 대화 변수로 주문 단계나 선택 항목을 따로 관리할 수 있습니다.
{% endhint %}

## 설정 방법

<figure><img src="../../../.gitbook/assets/image (18).png" alt="" width="366"><figcaption></figcaption></figure>

### 1. 대상 변수 선택

값을 변경할 대화변수를 선택합니다.

***

### 2. 연산 방식

변수의 값을 어떻게 변경할지 선택합니다. 4가지 방식이 있습니다.

| 연산       | 설명                                            | 예시                                   |
| -------- | --------------------------------------------- | ------------------------------------ |
| **덮어쓰기** | 기존 값을 지우고 새 값으로 교체합니다                         | `"홍길동"` → `"김철수"`                    |
| **값 지정** | 변수에 값을 설정합니다                                  | 빈 변수에 `"서울"` 지정                      |
| **누적**   | 기존 값에 새 값을 이어 붙입니다. 배열이면 항목이 추가되고, 숫자면 더해집니다  | `["사과"]` + `"바나나"` → `["사과", "바나나"]` |
| **지우기**  | 값을 비웁니다. 문자열은 `""`, 숫자는 `0`, 배열은 `[]`로 초기화됩니다 | `"홍길동"` → `""`                       |

{% hint style="info" %}
#### 어떤 연산을 써야 할까요?

* 이전 값이 필요 없고 새 값으로 바꾸고 싶다 → **덮어쓰기**
* 배열에 항목을 하나씩 쌓거나, 숫자를 합산하고 싶다 → **누적**
* 다음 루프를 위해 변수를 깨끗이 비우고 싶다 → **지우기**
{% endhint %}

***

### 3. 값 지정 방식

새로 넣을 값을 어디서 가져올지 선택합니다.

| 방식   | 설명              | 사용 예시                 |
| ---- | --------------- | --------------------- |
| 고정 값 | 직접 입력한 값을 사용    | 단계를 `"완료"`로 변경할 때     |
| 변수   | 다른 노드의 결과 값을 사용 | LLM의 출력을 대화 변수에 저장할 때 |

***

### 4. 여러 변수 동시 변경

우측의 + 버튼을 클릭하여 하나의 변수 할당 노드에서 여러 변수를 동시에 변경할 수 있습니다.&#x20;

항목을 추가하여 각각 다른 변수에 다른 연산을 적용할 수 있습니다.

***

## 출력 변수

변수 할당 노드는 별도의 출력 변수를 생성하지 않습니다. 대신 대상 변수의 값을 직접 변경합니다.

***

## 활용 예시

#### 1. 대화 변수: 사용자의 감정에 맞춰 응대하기

고객 상담 챗봇에서 대화가 진행될수록 사용자의 감정 상태를 파악하고, 그에 맞게 응대 톤을 조절합니다.

**어떻게 동작하나요?**

> 사용자: "아까부터 계속 안 되는데요"\
> → LLM이 감정을 분석: "불만"\
> → 대화 변수 `user_mood`에 "불만" 저장\
> → 다음 응답부터 LLM에 "사용자가 불만 상태이니 더 공감하며 정중하게 응대하세요"라는 지시를 함께 전달

**구성**:

```
시작 → LLM (감정 분석) → 변수 할당 (user_mood 업데이트) → LLM (응대) → 답변
```

* 대화 변수: `user_mood`(문자열) = `"보통"`
* 첫 번째 LLM 노드: 사용자의 메시지를 읽고 감정을 "만족 / 보통 / 불만 / 화남" 중 하나로 판단
* 변수 할당 노드: 판단 결과를 `user_mood`에 덮어쓰기
* 두 번째 LLM 노드: `user_mood` 값을 프롬프트에 포함하여 톤을 조절

매 대화마다 감정을 다시 분석하므로, 사용자의 기분이 풀리면 자연스럽게 응대 톤도 돌아옵니다.

***

#### 2. 대화 변수: 대화 중 정보를 하나씩 모으기

쇼핑 챗봇에서 사용자가 대화 중 관심을 보인 제품들을 차곡차곡 모아두었다가, 마지막에 한꺼번에 추천합니다.

**어떻게 동작하나요?**

> 사용자: "노트북 좀 볼게요" → `interested_items`에 "노트북" 추가\
> 사용자: "마우스도 필요해요" → `interested_items`에 "마우스" 추가\
> 사용자: "이제 추천해주세요" → 지금까지 모인 목록을 기반으로 맞춤 추천

**구성**:

* 대화 변수: `interested_items`(문자열 배열) = `[]`
* LLM 노드가 사용자 메시지에서 제품 키워드를 추출
* 변수 할당 노드에서 `interested_items`에 추가(append)
* 추천 단계에서 `interested_items` 전체를 LLM에 전달

배열의 **추가(append)** 연산을 사용하면 이전 값을 덮어쓰지 않고 계속 쌓을 수 있습니다.

***

#### 3. 루프 변수: 상세 리서치

사용자의 질문에 대해 한 번의 검색으로 부족하면, AI가 스스로 "아직 부족한 부분"을 찾아내고 추가 검색을 반복합니다.

**어떻게 동작하나요?**

> 사용자: "재택근무 정책이 어떻게 되나요?"\
> 1회차: 지식 검색 → "재택근무 신청 방법" 발견 → 루프 변수에 저장\
> → LLM: "신청 방법은 찾았지만, 대상자 기준과 장비 지원 정보가 부족합니다"\
> 2회차: "재택근무 대상자"로 재검색 → 결과 추가 저장\
> → LLM: "대상자 기준도 찾았습니다. 충분합니다" → 루프 종료\
> 최종: 수집된 정보를 종합하여 답변

**구성**:

```
시작 → 루프 → [지식 검색 → LLM (평가 + 다음 검색어 생성) → 변수 할당] → LLM (종합 답변) → 답변
```

* 루프 변수: `collected`(문자열 배열) = `[]`, `enough`(불리언) = `false`
* 지식 검색 노드: 검색어로 자료 탐색
* LLM 노드: 지금까지 모인 자료(`collected`)를 보고, 충분한지 판단. 부족하면 다음에 찾아볼 검색어를 제시
* 변수 할당 노드: 검색 결과를 `collected`에 추가(append). 충분하다면 `enough`를 `true`로 변경
* 중단 조건: `enough` = `true`

핵심은 **루프 변수에 결과를 누적하고, 그 누적된 내용을 다시 LLM에 전달**하는 것입니다. LLM이 "지금까지 모인 것"과 "아직 부족한 것"을 매번 비교하므로, 점점 더 완성도 높은 자료가 모입니다.
