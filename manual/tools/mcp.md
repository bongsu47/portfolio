# MCP

## MCP의 개념

***

MCP(Model Context Protocol)를 한 줄로 설명드리면, GenAI가 다양한 프로그램을 쉽게 연동하여 행동할 수 있도록 만든 표준 통신 방식입니다.

MCP는 Sonnet, Haiku 등의 GenAI 모델을 서비스하고 있는 엔트로픽(Antropic)에서 개발하였습니다. 경쟁사인 OpenAI에서도 공식적으로 지원할 만큼 현재 업계에서 사실상의 표준(de facto standard)로 받아들여지고 있습니다. 앤트로픽의 MCP 서버 페이지를 보면 이미 엄청 많은 MCP 서버 코드가 공개된 것을 볼 수 있습니다.

{% embed url="https://github.com/modelcontextprotocol/servers" %}

<figure><img src="../../.gitbook/assets/image (147).png" alt=""><figcaption><p>출처: <a href="https://norahsakal.com/blog/mcp-vs-api-model-context-protocol-explained/">What is Model Context Protocol (MCP)?</a></p></figcaption></figure>



MCP에 대해서 좀 더 쉽게 설명드리면, ChatGPT나 Claude 같은 GenAI가 구글 캘린더, 슬랙 등과 같은 업무 프로그램들과 자연스럽게 연결될 수 있게 해주는 '공통 연결 방식'이라고 생각하시면 됩니다. 마치 USB 케이블 하나로 프린터든 마우스든 외장하드든 어떤 기기나 연결할 수 있는 것처럼, MCP 표준에 맞게 설계된 서버라면 GenAI와 쉽게 연결되어 사용될 수 있게 되는 거죠.

미소 사용자 관점에서 말씀드리면, 미소의 앱에서 사용할 수 있는 도구인데 그 도구가 MCP라는 방식으로 구현된 것이다 라고 이해할 수 있습니다.

이렇게 설명드리면 “MCP 없이도 기존 도구를 앱에 연동하여 다양한 기능들을 잘 처리 했는데?”와 같이 기본 도구와 MCP가 무슨 차이가 있는가? 라는 의문이 들 수 있습니다.

기본 도구 대비 MCP를 사용하는 방식의 가장 큰 장점 2가지는 규격화와 확장성입니다.

### 규격화

<figure><img src="../../.gitbook/assets/image (148).png" alt=""><figcaption></figcaption></figure>

MCP 이전에는 AI에게 특별한 능력을 부여할 때는 주로 HTTP API라는 방식을 사용했습니다. 실제로 현재 미소에 정의된 다양한 도구은 대부분HTTP API 형식으로 서비스를 제공해 주는 서버와 통신하는 코드형태로 미소 내부에 구현되어 있습니다.

예를 들어 미소의 “위키백과” 도구는 위키백과에서 제공해 주는 [HTTP API](https://en.wikipedia.org/api/rest_v1/#/)를 이용하여 작성한 코드로 구현되어 있습니다.

위키백과의 API 문서에는 이렇듯 다양한 기능들을 정의하고 있고 각각의 기능들을 사용하는 방법에 대해서 서술하고 있습니다. 위키백과와 관련된 기능을 미소의 도구로 구현하기 위해서 위의 API 문서에 기반하여 이를 사용하는 코드를 미소 도구로 구현되어 있습니다.

<figure><img src="../../.gitbook/assets/image (149).png" alt=""><figcaption><p>위키백과에서 제공되는 API 목록</p></figcaption></figure>

GenAI 앱에 외부 서비스를 연동했다고 하면 결국 그 서비스에서 할 수 있는 어떤 동작을 요청하게 될 텐데요. 이때 가능한 동작들을 규정해둔 것이 바로 MCP의 도구(미소의 도구와는 다른 개념)입니다. MCP의 가장 강력한 장점은 바로 이 도구(tool, 미소의 도구와는 다른 MCP에 정의된 개념)입니다.

미소 에이전트에게 위의 MCP를 연동하게 되면 에이전트는 이 도구들을 조합해서 사용자의 요청을 처리하게 됩니다. MCP의 장점 중 하나는 바로 이 “도구”들이 이미 규격화되어 있으며, 에이전트에게 간단한 프롬프팅만 추가하여도 스스로 사용 가능한 도구를 알아보고 선택해서 쓸 수 있다는 점입니다. MCP 이전에는 위키백과 도구의 예처럼 이 각각의 MCP 도구들로 정의된 기능들이 딱히 규격화되어 있지 않았습니다.

노션 MCP 서버를 NOTION MCP와 같은 이름으로 미소에 등록한 뒤, 에이전트에 세팅하여 “사용자의 노션과 관련된 요청에 대해서 NOTION MCP를 이용하여 명령을 처리하라”라고 입력하면 에이전트는 여러 MCP 도구 중 사용자의 명령을 수행할 수 있는 적절한 것을 찾아 임무를 수행합니다. 필요에 따라 각 도구의 사용법을 프롬프트에 설명해 줄 수도 있습니다.

MCP를 사용하지 않는 경우라면 기능 각각에 대해서 모두 미소에 도구로 등록 요청을 하거나, 워크플로우에서 코드 노드를 이용하여 각각 구현한 뒤 미소 도구로 구현하는 번거로운 과정을 수행하여야 할 것입니다.

### 확장성

미소로 GenAI 서비스를 구현해야하는데 꼭 필요한 기능이 있다고 가정해 보겠습니다. 예를 들어 사용자의 질문에 대해서 지도를 검색해서 답변해 주어야 하는 상황이라고 해보겠습니다. 이런 경우 지도 도구가 필요합니다. 미소에 등록되지 않은 도구이므로 2가지의 방법이 있습니다.

1. 미소 관리자에게 지도 관련 도구 등록을 요청한다.
2. 위에서 설명한 HTTP API 방식으로 코드 노드를 이용하여 워크플로우를 구현하고 이를 워크플로우 도구로 만들어서 사용한다.

1번은 시간이 오래 걸리고 2번은 개발 지식이 없으면 할 수 없는 방법입니다. 하지만 지도 서비스를 제공해 주는 MCP 서버가 있다면 어떨까요? 뒤에서 설명드릴 간단한 과정을 통해서 MCP 서버를 미소에 등록하고 에이전트에 연결해 준뒤 간단히 MCP에 대해 설명하는 프롬프트만 추가해 주면 끝입니다.

에이전트에게 “사용자의 질문에 답변하기 위해 지도 검색이 필요한 경우 지도 MCP를 사용하라” 정도의 프롬프트만 입력하면 됩니다.

## MCP 상상해 보기

***

미소에서 기존의 도구를 사용하는 것 대비 MCP를 이용하는 것의 가장 큰 장점은 직접 등록해서 사용할 수 있다는 점입니다.

내가 원하는 기능을 제공해 주는 MCP 서버가 있다면 (회사의 보안 정책을 위반하지 않는 선에서) 얼마든지 미소에 등록하고 미소 앱에서 사용할 수 있습니다. MCP 서버들은 표준화된 MCP 규약을 따르기 때문에 등록 방법은 대동소이 합니다.

만약 우리 회사에서 사용중인 다양한 서비스들을 활용할 수 있는 MCP 서버를 미소에 연동할 수 있다면 어떨까요?

예를 들어 팀즈 채팅 MCP 서버를 우리회사 미소에 등록한다면 에이전트에게 간단한 자연어 명령으로 채팅방을 만들 수도 있고, 여러 채팅방에 동일한 공지를 동시에 작성할 수도 있을 것입니다.

만약 우리 회사 내부의 데이터를 조회할 수 있는 MCP 서버가 있다면 어떨까요?

발전 도메인의 회사라면 “과거 LNG1호기에서 발생한 고장 이력을 최근 1달치 조회하고 그 중에서 연료의 품질과 관련된 사례들만 골라서 알려줘”와 같은 질문에 대해서 답을 해주는 에이전트를 간단히 구현할 수 있게됩니다.

물론 MCP 서버를 사용하지 않고도 구현은 가능합니다. 하지만 훨씬 어렵고 복잡하며 아마도 개발자가 필요할 겁니다. 과거 기록을 조회할 수 있는 MCP 서버가 사내에 구현되어 있다면 해당 MPC 서버를 미소에 등록해 두고 이런 기능을 하는 에이전트를 손쉽게 누구나 구현할 수 있습니다. MCP 서버를 미소에 등록하고 에이전트에 연결하여 MCP의 사용법만 간단히 설명해 주면 끝입니다.

## 미소에서 제공하는 MCP 서버 인증 방식

***

앞서 MCP 예제에서는 서버와 인증처리 없이 동작하였습니다. 여기서 API키는 해당 MCP서버에 접근할 때 접근할 권리가 있는 사용자라는 것을 증명하기 위해서 사용하는 비밀번호와 같은 것이라고 이해하시면 됩니다.

URL에 API 키를 보내는 방식은 보안 관점에서 좋은 방식이 아닙니다. 실제 업무 환경에서 사용될 MCP 서버는 아래에서 설명하는 방식으로 보안을 강화하여 사용하는 것이 바람직합니다.

미소에서 MCP 서버를 연결하는 경우 제공하는 서버 인증방식은 총 6가지입니다. 각각의 방식을 간단히 설명하면 아래와 같습니다.



<figure><img src="../../.gitbook/assets/image (406).png" alt="" width="375"><figcaption></figcaption></figure>





* 없음
  * 인증이 없거나 URL에 API 키를 입력하는 경우
* Basic Auth
  * 사용자 ID와 비밀번호를 함께 보내 인증하는 가장 단순한 방식입니다.
* Bearer Token
  * 서버에서 발급받은 토큰(디지털 열쇠)을 사용해 인증하는 방식입니다.
* 사용자 정의
  * 인증 방식과 키 값을 시스템에서 자체적으로 정하는 방식입니다.
  * (MCP) 서버 개발자가 임의로 지정한 값으로 인증하는 방식입니다.
* OAuth 2.0
  * 구글, 네이버 등 외부 서비스를 통해 로그인하거나 권한을 위임받는 표준 인증 방식입니다.
  * 보안성이 높고 다양한 앱 간 연동에 적합합니다.
* 사용자 개별 토큰
  * Microsoft Azure 등 SSO 로그인으로 인증한 사용자의 토큰을 MCP 서버에 자동으로 전달하는 방식입니다.
  * 일반적으로 해당 옵션은 보이지 않습니다. 어드민이 고객사 설정에서 `MS Azure 로그인` 을 `OAuth` 타입으로 활성화하고, `추가 권한 요청 활성화` 를 해야만 보입니다.



결론부터 말씀드리자면 MCP 서버가 어떤 인증 방식을 사용하는지 여부 그리고 해당 인증 방식에서 입력해야 하는 값들을 MCP 서버 제공자에게 받은 뒤 인증 방식을 선택 후 해당 값들을 입력하면 MCP 서버는 잘 동작합니다.&#x20;



아래 문서에서는 단순히 MCP 서버 연결 이상으로 각각의 인증 방식들에 대한 간략한 소개와 해당 인증 방식에서 입력해야 하는 값들이 어떤 의미를 지니고 있는지를 알아봅니다.



### **Basic Auth**

**Basic Authentication**은 가장 단순한 방식의 인증 방법으로, 사용자의 **아이디(username)** 와 **비밀번호(password)** 를 함께 전송하여 서버가 해당 사용자를 인증하도록 하는 구조입니다.



<figure><img src="../../.gitbook/assets/image (211).png" alt=""><figcaption></figcaption></figure>

#### **Username (사용자 이름)**

*   **설명**:

    서비스를 사용하기 위해 등록한 **사용자 계정 ID**입니다.

    이메일 주소 형식일 수도 있고, 시스템에서 부여한 고유 아이디일 수도 있습니다.
*   **예시**:

    ```
    api_user_123
    ```

    또는

    ```
    user@example.com
    ```

#### **Password (비밀번호)**

*   **설명**:

    위 username과 함께 사용되는 **계정의 인증 비밀번호**입니다.

    해당 계정을 사용해 로그인할 때 입력하는 패스워드와 동일하거나,

    별도로 발급된 **API 전용 비밀번호**일 수 있습니다.
*   **예시**:

    ```
    p@ssw0rd!123
    ```



### **Bearer Token**

Bearer Token 인증은 **외부 시스템에 안전하게 접속하기 위해 토큰(일종의 디지털 열쇠)**&#xC744; 사용하는 방식입니다.

<figure><img src="../../.gitbook/assets/image (212).png" alt=""><figcaption></figcaption></figure>

#### **인증 헤더 키**

> HTTP 헤더는 웹이나 시스템 간 통신에서, 요청 또는 응답에 대한 "부가적인 정보"를 담는 부분입니다.
>
> 여기서 입력되는 키와 값들은 미소 서버가 HTTP 헤더에 담아서 MCP 서버에 전달하여 인증을 처리하게 됩니다.

*   **설명**:

    서버에 "지금 보내는 요청은 인증된 사용자로부터 온 것"임을 알려주는 HTTP 헤더의 이름(키)입니다.

    대부분의 경우 이 키의 이름을 **`Authorization`** 으로 고정해서 사용합니다.

#### **인증 값**

*   **설명**:

    실제 인증 수단이 되는 토큰(Token)을 서버에 전달하는 값입니다. 보통 "Bearer"라는 키워드 뒤에 발급받은 **Token**을 붙여서 전송합니다. 즉 미소 서버가 아래와 같은 값을 HTTP 헤더에 담아서 해당되는 MCP 서버에 전송하여 적법한 사용자임을 인증합니다.
*   **형식 예시**:

    ```
    Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
    ```
*   **예시**:

    | 항목      | 값                     |
    | ------- | --------------------- |
    | 인증 헤더 키 | `Authorization`       |
    | 인증 값    | `Bearer ABC123XYZ456` |
* **주의사항**:
  * 토큰 값은 민감한 정보입니다. 절대 외부에 공유하거나 복사해선 안 됩니다.



### **사용자 정의**

**사용자 정의 인증**은 시스템에서 자체적으로 정한 방식으로 접근 권한을 부여하고 사용자 신원을 확인하는 인증 방식입니다. 즉, 표준 인증 방식(Basic, Bearer 등)을 사용하지 않고, (MCP) 서버 제공자가 **내부 규칙에 따라 인증 값을 구성**합니다.

<figure><img src="../../.gitbook/assets/image (213).png" alt=""><figcaption></figcaption></figure>

#### **인증 헤더 키**

*   **설명**:

    이 값은 서버에게 "이 요청은 인증 정보가 포함되어 있다"는 사실을 알려주는 HTTP 헤더의 **이름**입니다.

    표준 방식에서는 `Authorization`이 주로 사용되지만, 사용자 정의 인증에서는 (MCP)서버 제공자가 임의로 지정할 수 있습니다.
* **어떻게 입력하나요?**
  * 서버에서 요구하는 키 이름을 그대로 입력해야 합니다. (MCP 서버 담당자에게 문의)
  * 예를 들어 `X-API-KEY`, `X-CUSTOM-AUTH`, 또는 `Authorization` 등을 사용할 수 있습니다.
*   **예시**:

    ```
    X-API-KEY
    ```

    또는

    ```
    Authorization
    ```

#### **인증 값**

*   **설명**:

    이 값은 실제 인증 정보를 담고 있으며, 서버가 사용자를 식별하고 요청을 허용할 수 있도록 **비밀 키나 토큰 등으로 구성된 문자열**입니다.
* **입력 방법**:
  * 어떤 형식으로 입력해야 하는지는 시스템마다 다르므로, 보통 서버 제공자에게 문의하여야 합니다.

#### 입력 예시 정리

| 항목      | 설명                  | 예시                                            |
| ------- | ------------------- | --------------------------------------------- |
| 인증 헤더 키 | 인증 정보를 담는 헤더의 이름    | `X-API-KEY`, `Authorization`, `X-CUSTOM-AUTH` |
| 인증 값    | 실제 인증 데이터 (토큰, 키 등) | `abc123xyz456` 또는 `CustomAuth key=..., ...`   |

#### 참고 사항

* 사용자 정의 인증은 **회사나 특정 API 제공자에서 자체적으로 설계한 인증 방식**이므로, 반드시 사전에 제공된 문서나 가이드를 참고하여 정확히 입력해야 합니다.
* 이 방식은 표준 방식이 아닙니다. 반드시 적절한 보안성 검토 후 사용하시는 것이 바람직합니다.



### **사용자 개별 토큰**

#### 설명

* 사용자가 Microsoft 계정으로 로그인한 경우, 해당 사용자의 액세스 토큰이 MCP 서버 호출 시 자동으로 Bearer 헤더에 포함됩니다.
* 이를 통해 사용자별 권한에 따른 개인화된 서비스를 제공할 수 있습니다. 예를 들어 사용자의 OneDrive, Teams, Outlook 등 Microsoft 365 서비스에 접근하는 MCP 서버를 구현할 수 있습니다.



#### 전제 조건

* 어드민이 고객사 설정에서 `MS Azure 로그인` 을 `OAuth` 타입으로 활성화하고, `추가 권한 요청 활성화` 를 해야만 해당 옵션이 보입니다.



#### Provider

* 현재 `Microsoft` 만 지원됩니다. 향후 추가 예정입니다.
